## 第7页：Planner-Executor-Replanner闭环机制

### 闭环控制流程

```
┌─────────┐    ┌─────────┐    ┌───────────┐
│ Planner │ →  │Executor │ →  │Replanner │
│ (规划)  │    │ (执行)  │    │ (重规划) │
└─────────┘    └─────────┘    └───────────┘
     ↑                               │
     └───────────────────────────────┘
              反馈调整回路
```

### 核心机制说明

| 角色 | 功能 | 关键能力 |
|------|------|----------|
| Planner | 任务规划 | 将运维指令转化为结构化任务序列 |
| Executor | 任务执行 | 调用原子工具集(QueryMetricsTools等) |
| Replanner | 动态调整 | 对比预期目标与实际观测值，修正执行路径 |

### 示例场景（微服务接口响应超时）
- **Plan**：排查网络连通性 → 检查资源水位 → 分析下游依赖
- **Execute**：调用QueryMetricsTools获取CPU/内存指标
- **Replan**：若资源正常，转向数据库层检查；生成新排查方向

## 问题3：你的创新点是什么

### 评审老师提问
> 你的系统有什么创新点？和现有运维平台有什么不同？

### 参考回答

感谢老师。我的系统主要有三个创新点：

**第一个创新点是Planner-Executor-Replanner闭环架构**
现有运维平台大多是预设规则的，遇到问题按固定流程执行，如果中间出错了就直接失败。我的系统引入了"自省-修正"机制：当Executor执行失败时，Replanner会分析原因，重新生成排查路径。比如查CPU指标返回空值，系统会自动判断是时间窗口问题，先校准时间再重新查询。这个闭环机制让系统具备了动态适应能力。

**第二个创新点是多Agent协作**
系统设计了Supervisor、Planner、Executor三种Agent角色，分工明确、协同工作。Supervisor负责调度，Planner负责规划，Executor负责执行，实现了运维流程的感知-决策-执行一体化，而不是简单的问答匹配。

**第三个创新点是RAG知识库的智能分片策略**
我设计了三层分片策略：按章节拆分、按段落切分、重叠窗口保证语义连贯。这样既保持了知识点的完整性，又提高了检索精度，解决了传统文档切片容易丢失上下文的问题。

简单总结：现有平台是"规则驱动"，我的系统是"智能驱动"，能自主规划、动态调整、持续学习。

## 问题5：请介绍一下Planner-Executor-Replanner

### 评审老师提问
> 你提到的Planner-Executor-Replanner架构，能简单解释一下吗？

### 参考回答

感谢老师。Planner-Executor-Replanner是我系统的核心架构，简单来说就是"规划-执行-重规划"三步走：

**第一步：Planner规划**
当系统收到一个运维问题，比如"数据库响应慢"，Planner会分析问题并生成排查步骤：先查日志看看有没有报错 → 再检查连接池状态 → 最后查CPU指标。这就像运维人员脑子里先想好排查思路。

**第二步：Executor执行**
Executor按照Planner制定的步骤，调用具体工具执行。比如调用日志查询工具获取最近一小时的错误日志，调用指标查询工具获取CPU使用率。执行结果返回给系统。

**第三步：Replanner重规划**
这是关键环节。Replanner会检查执行结果，判断问题是否已经找到。如果证据不够，比如查CPU发现正常，Replanner会调整策略：CPU没问题，那可能是数据库慢查询，重新规划查数据库指标。

这个过程是循环的，直到找到根本原因或达到最大轮次（我设置的是8轮）。大多数问题在3-5轮内就能解决，最终生成分析报告。

用一句话总结：Planner定计划，Executor干活，Replanner看结果决定要不要改计划。

感谢老师。论文第7章我总结了三个主要局限性：

**第一个是数据规模有限**
目前系统用的是模拟运维数据和少量开源文档，真实生产环境日志每天可能是TB级别，向量检索和模型处理超长上下文的性能还需要进一步验证。改进方向是引入Redis缓存层，优化高频知识的检索效率。

**第二个是工具集只读限制**
现在的工具只能查日志、查指标，不能执行写操作（比如重启服务、调整配置）。运维场景很敏感，贸然自动执行写操作风险很大。改进方向是设计安全沙箱，加入人工审核流程，逐步开放可控的写操作。

**第三个是规划稳健性问题**
遇到非常规故障时，大语言模型可能会生成无效的排查路径，导致效率低甚至死循环。改进方向是引入运维领域专用模型，或者结合传统规则引擎，提高复杂场景的决策能力。

如果后续继续做，我会先解决性能瓶颈问题，再考虑安全机制和领域模型优化，让系统更接近生产可用状态。

## 问题8：RAG是什么，你的系统怎么用RAG

### 评审老师提问
> 你提到RAG技术，能解释一下RAG是什么？在你的系统中起什么作用？

### 参考回答

感谢老师。RAG是Retrieval-Augmented Generation，中文叫"检索增强生成"。

**简单解释RAG原理**
传统大语言模型直接回答问题，可能会"胡编乱造"，因为它不知道你的企业运维文档内容。RAG的做法是：先从知识库检索相关文档，把检索结果和问题一起送给大模型，让模型根据真实资料回答。这样回答更准确，不会乱编。

**我的系统怎么用RAG**
系统启动时，我把运维文档（比如《CPU高使用率处理方案》）上传，系统会：
1. 把文档切成800字符的小片段
2. 用嵌入模型把每个片段转成向量（1024维数字）
3. 存到Milvus向量数据库

用户提问时，系统会：
1. 把问题也转成向量
2. 在Milvus中找和问题向量最相似的3个文档片段
3. 把这3个片段作为参考资料，送给大模型生成回答

**RAG在我的系统中的作用**
运维知识库让系统具备了"专业知识"，用户问"内存泄漏怎么排查"，系统不是凭空回答，而是根据上传的运维手册给出处理步骤。这就是知识沉淀和传承的价值。

## 问题9：多Agent和单Agent有什么区别

### 评审老师提问
> 你说的是多Agent协作，那和多Agent和单个Agent有什么区别？为什么要用多个Agent？

### 参考回答

感谢老师。单Agent和多Agent的区别主要在复杂任务的处理能力上。

**单Agent的局限**
如果只用一个Agent处理运维问题，它需要同时做三件事：分析问题、决定调用什么工具、判断结果够不够。任务复杂时，Agent容易混乱，而且每一步都依赖上一步的结果，不能并行处理。

**多Agent的优势**
我的系统设计了三个专业Agent：
- Planner专注规划，不关心具体怎么执行
- Executor专注执行，不关心整体策略
- Replanner专注判断，不关心具体操作

每个Agent职责单一，逻辑清晰。更重要的是，Planner可以在Executor执行的同时思考下一步，Supervisor可以协调多个Executor并行执行不同任务。

**实际例子**
比如排查一个复杂故障，单Agent要顺序执行：查日志→等结果→分析→查指标→等结果→分析...每步都要等。多Agent协作时，Planner可以一次规划多个查询任务，Executor并行执行，效率更高。

简单说：单Agent像一个人从头做到尾，多Agent像团队分工合作，复杂任务处理更高效。
